= Module 3: Developer workflow without developer friction

== Introduction

* By this point in the workshop, you have established strong foundations of trust across the software supply chain. In Module 1, you made software composition visible by ingesting and analyzing SBOMs. In Module 2, you ensured that build artifacts are signed, traceable, and verifiably produced by your CI pipeline.

* The next challenge is not technical, it is operational: how do you apply these controls consistently across teams without slowing developers down?

* This is where link:https://platformengineering.org/blog/what-is-platform-engine[*platform engineering*^] comes in. Platform engineering teams build link:https://internaldeveloperplatform.org/what-is-an-internal-developer-platform/[*Internal Developer Platforms (IDPs)*^] that bring together tools, workflows, and security controls into a single, self-service experience for developers. Instead of relying on manual processes, security and best practices are embedded directly into the platform.

* In this workshop, the IDP is *Red Hat Developer Hub (RHDH)*. It provides a centralized interface for developers to discover services, access documentation, and use predefined workflows that align with organizational standards.

* Key capabilities of *RHDH* include:
+
. *Software Catalog* – a centralized view of services, components, and ownership  
. *TechDocs* – documentation integrated alongside code  
. *Search* – unified discovery across services and documentation  
. *Software Templates* – reusable, standardized workflows for creating new applications  
. *Plugins* – integrations with CI/CD, Kubernetes, security tools, and more  

image::RHDH_Core_Capablties.png[Red Hat Developer Hub Core Capabilities,link=self,window=blank]

* One critical challenge remains: how do we prevent unverified or unsafe images from reaching production?
* Security controls only scale when developers do not have to think about them. If teams must manually assemble pipelines or decide when policies apply, trust quickly erodes.
* This is where *Software Templates*, also known as *Golden Paths*, come in. They define opinionated, secure workflows that embed the guardrails established in earlier modules by design, including *SBOM* generation and analysis, code and image signing, vulnerability scanning, and *Enterprise Contract* policy checks.

NOTE: *Enterprise Contract* is the policy engine that governs whether an application can be promoted through the delivery pipeline. It is based on the upstream open-source project link:https://conforma.dev/[*Conforma*^], which provides policy-driven verification of software supply chain evidence such as *SBOMs*, vulnerability scan results, image signatures, and build attestations.

* By the end of this module, you will see how *Red Hat Developer Hub (RHDH)* turns these trusted building blocks into repeatable, production-ready *Golden Paths*.
* In this module, you will take on the role of a platform engineer, preparing *RHDH* for developer use by configuring access, enabling capabilities, and defining standardized workflows.
* Every application starts secure, every promotion is governed, and teams can move quickly without bypassing trust.

.Optional Deep Dive: How the Platform Works Together
[%collapsible]
====

You’ve now completed this module. The section below is optional and explains how this module introduced three key control layers in the platform. Each layer is responsible for a different aspect of how the developer experience is delivered and enforced.

Dynamic Plugins (Platform Capabilities)

In *Red Hat Developer Hub*, plugins are not baked into the application at build time. Instead, they are enabled dynamically through configuration.

Plugins are defined in configuration (for example, in ConfigMaps or Helm values) under the `dynamic.plugins` section. Each plugin entry specifies:

- Whether the plugin is enabled or disabled  
- Where the plugin is loaded from (local package or OCI image)  
- Optional plugin-specific configuration  

For example:

[source, yaml]
----
dynamic:
  plugins:
    - disabled: false
      package: ./dynamic-plugins/dist/backstage-plugin-kubernetes
----

When this configuration is applied through GitOps:
- The platform loads the plugin at runtime  
- No rebuild of the Developer Hub image is required  
- Changes can be versioned, reviewed, and promoted across environments  

This means platform capabilities are controlled declaratively. Adding or removing functionality becomes a configuration change, not a rebuild process.

---

Devfiles (Development Environment Behavior)

A *Devfile* defines how a developer workspace is constructed and behaves.

A Devfile typically includes:

- *Projects*: source code repositories to be loaded into the workspace  
- *Components*: containers, tools, and dependencies that make up the runtime  
- *Commands*: actions such as build, run, and test  
- *Events*: lifecycle hooks that trigger commands at specific stages  

image::devfile_anatomy.png[Devfile Anatomy Diagram,link=self,window=blank]

In this module, multiple parts of the Devfile work together:

- Environment variables are injected into the workspace from a ConfigMap  
- A command (`init-git-config`) uses those variables to configure Git  
- A *postStart* event automatically runs that command when the workspace starts  

This is critical.

It means:
- Developers do not manually configure commit signing  
- Every workspace is initialized with the same behavior  
- Security controls are enforced as part of the environment lifecycle  

The Devfile becomes the enforcement point for developer experience. Instead of relying on instructions, the platform ensures the correct setup is applied automatically.

---

Golden Paths (Application Delivery Standardization)

Golden Paths are exposed in *Red Hat Developer Hub* through *Software Templates*.

A Software Template defines how an application is created and delivered. It is written in YAML and executed by the *Scaffolder*, which automates the creation process.

At a high level, a template includes:

- *Header*: metadata used to display the template in the UI  
- *Parameters*: inputs collected from the developer via forms  
- *Steps*: actions executed by the Scaffolder  
- *Output*: links or results shown after execution  

Example structure:

[source, yaml]
----
apiVersion: scaffolder.backstage.io/v1beta3
kind: Template
metadata:
  name: example-template
spec:
  parameters:
    - title: Application Details
      properties:
        name:
          type: string
  steps:
    - id: publish
      action: publish:gitlab
  output:
    links:
      - title: Repository
        url: ${{ steps.publish.output.remoteUrl }}
----

How this works in practice:

1. The developer selects a template in the UI  
2. The platform renders a form based on the *parameters* section  
3. The developer provides input  
4. The *Scaffolder* executes the defined *steps*  
5. The application, repository, and pipeline are created automatically  

Steps can:
- Create repositories  
- Register components in the catalog  
- Trigger pipelines  
- Apply security controls such as signing and scanning  

Golden Paths shift responsibility:

- Platform teams define the approved way of building and delivering software  
- Developers consume that capability through self-service  

This ensures that every application created:
- Follows the same structure  
- Uses the same delivery pipeline  
- Includes the same security and compliance controls  

---

Putting It Together

These three layers operate together:

- *Dynamic plugins* control what capabilities are available in the platform  
- *Devfiles* control how developer environments are configured and behave  
- *Software Templates (Golden Paths)* control how applications are created and delivered  

This creates a system where:

- Platform behavior is defined as code and managed through Git  
- Developer environments are automatically configured and consistent  
- Application delivery follows a standard, secure, and repeatable process
