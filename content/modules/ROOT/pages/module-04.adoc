= Module 4: Enforce Policy and Promote Safely

== Introduction


* In the previous modules, you established the foundations of a trusted software supply chain, from signing artifacts and generating SBOMs, to enforcing policy and verifying integrity.
* Now, you will see how all of these capabilities come together in a real developer workflow.

* In this module, you step into the role of a developer and use a Software Template to create and deliver a new application.
* From the developer’s perspective, this is a simple, self-service experience, launching a template and following a guided workflow.

* Behind the scenes, the platform takes care of everything:
+
. Provisioning repositories and environments
. Configuring CI/CD pipelines
. Enforcing security and compliance policies
. Managing secrets and integrations
+
* As a developer, you don’t need to worry about the underlying infrastructure or tooling.
* You are provided with a standardized, approved path to production, allowing you to focus on writing and delivering features.

* By the end of this module, you will see how the platform enables teams to move fast, while ensuring that every change is trusted, compliant, and production-ready by design.

---

== Step 1: Execute the Software Template

=== 1.1 Step introduction

* Up to this point, you have been working in the role of a platform engineer, setting up the foundations.
* Now we switch perspective and step into the role of a developer, using the *User1* account you created earlier.
* As a developer, your goal is simple, get a production-ready service up and running without dealing with manual setup overhead.
* This is where Software Templates, also referred to as Golden Paths, come in.
* They provide a self-service way to create applications that already align with platform standards, including repository structure, CI/CD pipelines, and security controls.
* Under the hood, this is powered by the link:https://backstage.spotify.com/docs/portal/core-features-and-plugins/scaffolder[*Backstage Scaffolder*^], which automates the setup based on platform-approved patterns.

---

=== 1.2 Run the Software Template

. Switch to the *RHDH* tab in the Showroom.
. Click *Home* in the left navigation menu.
. Scroll down to the *Explore Templates* section.
. Select *Securing a Quarkus Service Software Supply Chain (Tekton)*.

image::select_software_template.png[select_software_template,align="center",link=self,window=blank]

* You will now be guided through a four-step wizard.
* This workflow replaces manual setup and automatically applies platform standards defined by the template.

* In the first form, enter the application details.
* The default values are already provided, so keep them as-is and click *Next*.

image::software_template_step1.png[software_template_step1,align="center",link=self,window=blank]

* In the second form, you will see the container image registry configuration.
* In this workshop, container images are pushed to *Quay.io*.
* The values are already pre-filled, so click *Next*.

image::software_template_Step2.png[software_template_Step2,align="center",link=self,window=blank]

* In the third form, configure the source code repository settings for *GitLab*.
. Set *Verify Commits* to *Enabled*.
. Leave all remaining values unchanged.
* Click *Review* to continue.

image::software_template_step3.png[software_template_step3,align="center",link=self,window=blank]

* You will now see a summary of all parameters that will be passed to the *Scaffolder*.
* These values are used to:
+
. Create the Git repository and application skeleton
. Configure CI pipelines
. Register the component in the RHDH Software Catalog
+
* Review the values, then click *Create* to start the scaffolding process.

image::software_template_review.png[software_template_review,align="center",link=self,window=blank]

* After a short time, the scaffolder workflow will complete and each step will show a green checkmark.
* Click *Open Component in Catalog* to view your newly created application in Developer Hub.

image::open_component_in_catalog.png[open_component_in_catalog,align="center",link=self,window=blank]

* In this step, you used a Software Template to create a new service through a self-service workflow.
* Instead of raising tickets or manually setting up repositories and pipelines, the platform applied the required standards automatically.

=== Step 3: Explore the Tekton Pipeline

=== 3.1 Step introduction

* In this step, you will observe the CI pipeline triggered by your signed commit.
* This pipeline enforces security guardrails at every stage of the software delivery lifecycle.
* Instead of relying on manual checks, the platform automatically verifies identity, validates artifacts, and enforces policy before allowing promotion.
* Your goal here is to follow the pipeline and understand how trust is continuously validated.

---

* Switch back to *Red Hat Developer Hub* in *Showroom* and select the *CI* tab.
* You should see the pipeline execution in progress. Note that it may take a few minutes to complete.

image::secure_pipeline_in_progress.png[Secure Pipeline Execution in Progress,align="center",link=self,window=blank]

* While the pipeline runs, let’s walk through the key stages.

---

=== 3.2 Verify Commit

* This task verifies that the code change was made by a trusted identity.

* It checks that the commit was signed and can be linked back to the developer who made the change.

image::verify_commit.png[verify_commit,align="center",link=self,window=blank]

* Click the *verify-commit* task and review the logs.

image::Verify_Commit_Log.png[Verify Commit Task Log Output,align="center",link=self,window=blank]

* The output confirms:
+
. The commit was signed successfully
. The identity of the developer (*user1*) is verified
. The signature is valid and trusted
+
* This ensures that every code change can be traced back to a verified developer, establishing trust at the source.

---

=== 3.3 Build Container

* The *build-container* task builds the image from verified source code.
* During this step:
+
. The container image is created
. An SBOM is generated
. Attestations and signatures are attached
+
image::build_container_task.png[Build Container Task,align="center",window=blank]
+
* All artifacts are pushed to *Red Hat Quay*.

* Switch to the *Image Registry* tab and observe:
+
. Container image
. Signature
. Attestation
. SBOM
+
image::Generated_Artifacts_Registery.png[Generated Artifacts in Quay Registry,link=self,window=blank]

* These artifacts travel together, ensuring traceability.

image::Upload_SBOM.png[Upload SBOM Task,link=self,window=blank]

=== 3.5 Security Policy Enforcement (ACS)

* The next set of tasks uses *Red Hat Advanced Cluster Security (ACS)* to enforce security policies.

image::ACS_Image_Check_Task.png[ACS Image Check Task,align="center",window=blank]

* These checks cover:

* *Image Check*

+

== Step 5: Promote to Staging

=== 5.1 Step introduction

* In this step, you will promote your application using a Git tag.
* This simulates a real release process, where promotion is triggered intentionally rather than on every commit.
* The platform automatically runs a promotion pipeline that validates the release before allowing it to progress.
* This ensures that only trusted, compliant, and verified artifacts move forward in the software supply chain.

---

* If you are not already logged in to *GitLab*, switch to the *GitLab* tab in Showroom and log in using:

* *Username*:

----
{gitlab_root_user}
----
+
* *Password*:
+
[source, role="execute", subs="attributes"]
----
{gitlab_root_password}
----

---

=== 5.3 Create a Release Tag

. Navigate to *GitLab Home*.
+
image::gitlab_home_page.png[gitlab_home_page,align="center",link=self,window=blank]
+
. Open the repository *my-quarkus-tkn*.
+
image::my-quarkus-tkn_repo.png[my-quarkus-tkn_repo,align="center",link=self,window=blank]
+
. Go to *Code → Tags*.
+
image::click_code_tag.png[click_code_tag,align="center",link=self,window=blank]

image::create_tag.png[create_tag,align="center",link=self,window=blank]

---

=== 5.4 Explore the Promotion Pipeline

. Switch back to the *CI* tab in *Red Hat Developer Hub*.
. Locate and expand the *promote-to-stage* pipeline.

---

* The pipeline begins by preparing the release:

* *extract-destination-image-tag*
Resolves the release version from the Git tag.

* *gather-images-to-verify*
Identifies the previously built image and links it to the release, ensuring traceability.

---

=== 5.5 Verify Enterprise Contract

* The *verify-enterprise-contract* task validates the image before promotion.

image::verify-enterprise-contract.png[verify-enterprise-contract,align="center",link=self,window=blank]

. Open the task and review the logs.

* This validation ensures:

. *Provenance* — the image was built by a trusted pipeline
. *Identity* — the image is signed and linked to a verified identity
. *Integrity* — the image and its attestations have not been tampered with
. *Composition* — an SBOM is present and attached
. *Policy* — vulnerabilities and organizational rules are enforced

* Only images that pass all checks are eligible for promotion.

---

=== 5.6 Promote and Deploy

* Once validated, the pipeline promotes the image:

. *copy-image*
Retags the verified image for production (for example, *prod-v1.0*)

. *update-deployment*
Updates deployment configuration to use the promoted image

* Through GitOps integration, tools like *Argo CD* automatically deploy the updated version.

---

=== 5.7 Step summary

* You promoted your application using a controlled, versioned release.
* The platform enforced validation before allowing promotion.
* No manual approvals, no guesswork, policy-driven automation.

* The key takeaway:
+
*Promotion is no longer a risk point, it becomes a controlled, verifiable, and automated step in the delivery process.*
