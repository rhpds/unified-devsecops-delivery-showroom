== Introduction

Modern software delivery has accelerated dramatically, but with that speed comes a growing blind spot. Applications are no longer built from scratch, they are assembled from hundreds of open source libraries, third-party components, and transitive dependencies. As a result, many organizations struggle to clearly understand what is actually included in the software they release. When a vulnerability is disclosed or an incident occurs, the challenge is no longer just fixing the issue, it is first identifying whether you are affected at all.

To address this, organizations are adopting link:https://www.cisa.gov/sbom[*Software Bills of Materials*^], or *SBOMs*. An *SBOM* provides a structured inventory of everything that makes up an application, including all direct and indirect dependencies. However, generating an *SBOM* alone is not enough. The real value comes from being able to analyze, query, and continuously evaluate that data against evolving vulnerability intelligence.

This is where *Red Hat Trusted Profile Analyzer*, or *RHTPA*, plays a critical role. *RHTPA* ingests *SBOM* documents, builds a detailed graph of the software components they contain, and correlates that information with known vulnerabilities and security advisories. Instead of static reports, organizations gain a continuously updated and searchable view of their software composition, allowing them to understand risk, answer audit questions, and make informed decisions about their applications.

== Step 1: Configure identity and access management

* To begin establishing software transparency, we first install *RHTPA*. This allows organizations to build a trusted inventory of application components, helping them understand what is in their software and assess potential risk.

=== 1.1 Install and Setup RHTPA

* Navigate to the Showroom Terminal and click the run button to run the script that installs and sets up *RHTPA*.
+
[source, role="send-to-wetty"]
----
~/lab-assets/install-rhtpa.sh
----
+
* This script prepares the environment for *RHTPA* by configuring identity integration and applying the required platform resources.
. Behind the scenes, this step integrates *RHTPA* with *Red Hat Build of Keycloak (RHBK)*, which acts as the identity provider for the platform. *RHBK* manages users, roles, and access policies, ensuring that only authorized users and systems can interact with software composition data.
. As part of this setup, a dedicated realm is created along with the roles and permissions required by *RHTPA*. These define how different actors, such as developers, automation pipelines, or platform administrators, access and interact with *SBOM* data.
. Finally, the script deploys the *RHTPA* instance using its Custom Resource. This instructs the *RHTPA Operator* to provision the required components in the cluster, making the service available for *SBOM* ingestion and analysis.
* The script might take a few minutes to complete. Once the script has completed, you will see an output similar to the following screenshot:

Why is the vulnerability report empty?
If no vulnerabilities are displayed, this is expected in a newly deployed environment.

*Reason*: *RHTPA* relies on background importer jobs to retrieve vulnerability data from external sources such as the National Vulnerability Database (NVD) and GitHub Security Advisories. These datasets must be downloaded and processed before vulnerabilities can be correlated with SBOM components.

This initial synchronization can take some time to complete, especially in new environments.

.Optional Deep Dive: How does RHTPA process and secure SBOM data?
[%collapsible]
====

You’ve now completed this module. The section below is optional and provides a deeper look at how *RHTPA* ingests, secures, and analyzes SBOM data.


---

* *Identity and access control*

*RHTPA* does not manage users directly. Instead, it relies on *Red Hat build of Keycloak (RHBK)* as an external identity provider.

At a high level:

. A user or pipeline authenticates with *RHBK*
. *RHBK* issues an access token containing identity and permissions (scopes)
. The request is sent to *RHTPA* along with this token
. *RHTPA* evaluates the token to determine whether the operation is allowed

In this workshop:
. The *frontend* client is used by human users accessing the UI
. The *cli* client is used by pipelines to upload SBOMs programmatically

This model ensures that all access to SBOM data is controlled and tied to a specific identity.

---

* *SBOM ingestion and indexing*

When an SBOM is uploaded:

. *RHTPA* parses the SBOM document (for example, SPDX or CycloneDX)
. Every component, dependency, and relationship is extracted
. This data is stored as a graph representing the full software composition

This graph structure allows *RHTPA* to:
. trace dependencies across layers
. understand transitive relationships
. support complex queries across components

---

* *Continuous vulnerability correlation*

SBOM data alone is not enough. *RHTPA* continuously enriches this data using vulnerability intelligence.

This is done through *importers*:

. *CVE importer* retrieves vulnerability data from sources such as the CVE Project
. *GitHub Advisory importer* pulls data from GitHub Security Advisories
. These datasets are refreshed periodically (for example, daily)

Once importer data is available:

. *RHTPA* matches SBOM components against known vulnerabilities
. Correlations are created between components and advisories
. Results are made available in the *Vulnerabilities* view

This is why, in a fresh environment, the vulnerability tab may initially appear empty — the importer jobs need time to populate the dataset.

---

*Putting it all together*

*RHTPA* provides a continuous, identity-aware view of software composition:

. Identity ensures only authorized users and systems interact with the data  
. SBOM ingestion builds a structured inventory of components  
. Importers continuously enrich that inventory with vulnerability intelligence  

Together, this enables organizations to move from static SBOM files to a live, queryable, and auditable view of their software supply chain.
